Component Architecture & Data Flow

Before writing a line, read this tree until you understand who owns what. The golden rule: state lives in the lowest common ancestor that needs it.

App.jsx ← owns: expenses[], todos[] │ ├── CryptoWeatherWidget.jsx ← owns its own fetched data │ (no props from App — it fetches independently) │ ├── ExpenseTracker.jsx props: expenses, onAdd, onDelete │ ├── ExpenseForm.jsx props: onAdd │ └── ExpenseList.jsx props: expenses, onDelete │ └── TodoList.jsx props: todos, onAdd, onToggle, onDelete ├── TodoForm.jsx props: onAdd └── TodoItem.jsx props: todo, onToggle, onDelete
Why does App.jsx own the expense and todo arrays?

Because state lifting: if two sibling components need to share data, that data must live in their common ancestor. Here, ExpenseForm writes new expenses and ExpenseList reads them. They are siblings. Their parent (ExpenseTracker) is the right home — or App.jsx if you later want a grand total across everything.

CryptoWeatherWidget breaks this pattern intentionally — its data is completely self-contained and nothing else cares about it, so it owns its own state.

Tailwind v4 — One Thing to Know

Your project already has Tailwind v4 installed. It works differently from v3.

  • There is no tailwind.config.js to create. Configuration is done in CSS.
  • index.css already has @import "tailwindcss" — that one line is all the setup needed.
  • All utility classes (flex, grid, p-4, etc.) work identically to v3.
  • In v4, the md: breakpoint defaults to 768px — same as v3.

1.1
Semantic HTML Skeleton in App.jsx

The outer shell of the page uses meaningful HTML5 elements — not just divs.

Hint — What's the difference between <div> and <section>?

A <div> is purely structural — it carries no meaning. A <section> says "this is a standalone, thematically grouped piece of content." Screen readers and search engines care about this distinction. The rule: if your grouped content could have its own heading, it should be a <section>.

1.2
Responsive Three-Column Grid

The three sections should stack on mobile and sit side-by-side on desktop. This is done with a single container div around the three sections.

Concept — How does the Tailwind responsive prefix work?

Tailwind is mobile-first. A class without a prefix applies to all screen sizes. A class with md: applies at the md breakpoint (768px) and above.

So grid-cols-1 md:grid-cols-3 means: "1 column always, then override to 3 columns once the screen is ≥768px." You never need to write media queries by hand.

Common mistake — <button> type attribute

A <button> inside a <form> has a default type of "submit". This means even your "Delete" button will submit the form if you forget to set type="button" on it. Always be explicit: form submission button gets type="submit", all others get type="button".

Phase 2
Forms, State & Interactivity
Replace all hardcoded data with React state. Every widget should be fully interactive by the end of this phase.
2.1
Controlled Input — ExpenseForm

A "controlled input" is an input whose value is driven by React state, not by the DOM. This is the standard React way.

Concept — Why "controlled"? What happens without it?

Without a value binding, the input is "uncontrolled" — the DOM owns its value and React doesn't know about it. With value={state} + onChange, React owns the value. This is essential for: clearing inputs after submission, validating as the user types, and syncing input values with other parts of the UI.

Think of it this way: uncontrolled means "the DOM stores the truth." Controlled means "React stores the truth and the DOM just displays it."

Hint — Price input: text or number state?

The <input type="number"> DOM value is always a string (e.g. "12.50"). When you update state from e.target.value, you're storing a string. When you need to do math with it later, convert it: parseFloat(value). Store as string in state, convert to number at use-time.

2.2
Parent State — The Expense Array

The list of expenses is an array of objects. This is "complex state." It lives in the parent component.

Warning — Never mutate state directly

Do NOT do: expenses.push(newItem) then setExpenses(expenses). This mutates the original array in place. React compares object references, not deep values. Since the array reference didn't change, React sees no difference and will NOT re-render.

Do this instead: setExpenses([...expenses, newItem]). The spread operator creates a brand new array with all the old items plus the new one. New reference = React re-renders.

Hint — How to generate a unique ID

Date.now() returns the current timestamp in milliseconds — it's unique enough for a learning project. In production you'd use a UUID library, but this is fine here. Use it as the id of your new object.

2.3
State Lifting — Connect Form to Parent

The form lives in a child component. But the state lives in the parent. The parent passes a function down to the child so the child can trigger an update up.

Concept — State Lifting in one sentence

When a child needs to change its parent's state, the parent passes a callback function as a prop. The child calls that function with the new data. The parent updates state. React re-renders both. The child never touches the state directly — it only calls the function it was given.

2.4
Rendering the List with .map()

Replace the hardcoded items with a dynamic render of the expenses array.

Warning — Why you must never use array index as key

When items can be deleted or reordered, using the array index as key causes React to incorrectly reuse DOM nodes. Example: you have items [A, B, C] at indices [0, 1, 2]. You delete B. Now you have [A, C] at indices [0, 1]. React sees key=1 still exists and thinks C is still B — it may preserve old input values or incorrect DOM state.

Use a stable, unique ID from the data instead. That's why you added id to each expense object.

2.5
Delete an Expense with .filter()

Deletion is the mirror of addition: instead of adding to the array, you return a new array with one item removed.

Hint — How to wire the onClick with the item's id

The delete button needs to call the function with a specific id. If you write onClick={onDelete(expense.id)} — that calls the function immediately on render, not on click. You need an arrow function wrapper: onClick={() => onDelete(expense.id)}. The arrow function is created fresh on each render, but it closes over the specific expense.id of that row.

2.6
Running Total with .reduce()

Calculate the live total from the array. It should update every time an item is added or removed.

Concept — Derived state vs stored state

The total is a value you can always compute from the expenses array. Storing it in its own useState would be a mistake — you'd have to keep two pieces of state in sync, which is a common source of bugs. Instead, derive it: const total = expenses.reduce(...). Every time expenses changes and React re-renders, total is recalculated for free. This principle — "derive, don't duplicate" — is fundamental to thinking in React.

2.7
Build the Todo List (Same Pattern)

Apply everything you just learned to the Todo list. The key difference: todos have a completed boolean you can toggle.

Hint — The toggle pattern, step by step

You have an array of todos. You want to flip completed on the one with a specific id. The right tool is .map():

Loop through every item. For each one: if its id matches, return a new object that copies all existing properties with spread, but overrides completed with its opposite. If the id does not match, return the item as-is.

This creates a brand new array with a brand new object at the changed position. Everything else is untouched. React sees a new array reference and re-renders.

2.8
Conditional Styling on TodoItem

A completed todo should look visually different. This is conditional rendering — applying classes based on a boolean.

Hint — Template literal + ternary in className

Use a JavaScript template literal (backticks) to combine fixed classes with conditional ones. The ternary sits inside ${"$"}{}. Structure: fixed classes always apply, then a conditional expression adds or omits extra classes based on the boolean.

If the ternary has no "else" classes, use an empty string "" as the falsy branch to avoid printing undefined into the class list.

3.2
Set Up the Three Async State Variables

Every async data fetch needs exactly three pieces of state. This is the universal pattern.

Concept — Why three separate variables and not one object?

You could combine them into one object: { data, loading, error }. But three separate useState calls is cleaner for a beginner — each setter only touches one concern. The three-variable pattern is also exactly what the React useReducer pattern and libraries like React Query build on top of. You'll recognise it everywhere once you see it here.

3.3
Write the useEffect Hook

This is the most important step in Phase 3. Read every hint before you write.

Concept — Why useEffect? Why not just fetch at the top of the function?

React component functions run on every render. If you placed a fetch() call directly in the function body, it would fire an HTTP request on every single re-render — including the re-renders caused by the data arriving. You'd get an infinite loop.

useEffect is React's way of saying "run this code as a side effect, after rendering, and control exactly when it repeats." The empty [] dependency array means "the conditions for re-running never change," so it runs exactly once after the first render.

Warning — The empty array [] is not optional

If you omit the second argument entirely: useEffect(fn) — the effect runs after every render. Your fetch call would fire every time any state changes, creating an infinite loop as the arriving data triggers a re-render which triggers another fetch.

If you pass [someVariable] in the array: the effect re-runs whenever someVariable changes. An empty array [] has no variables that can change, so it only ever runs once.

3.4
Conditional Rendering — Loading, Error, Data

The widget's UI has three possible states. Show a different UI for each.

Hint — Early returns vs nested ternaries

For three distinct states, early returns are cleaner than nesting ternaries. Structure your component with conditional return statements at the top of your JSX: if loading, return the spinner; if error, return the error message; otherwise, fall through to the main return with the data.

This avoids deeply nested ? ... : ... chains that become hard to read. Early returns are the same technique used in Next.js server components.

Reference
React Concepts Quick Reference
Not instructions — these are the mental models behind what you just built. Read them when something doesn't feel right.

useState

Returns [value, setter]. Calling the setter schedules a re-render. The setter replaces state entirely — it does not merge. For the new value to depend on the old one, use the functional form: setState(prev => ...).

useEffect

Runs after rendering, not during. The second argument controls when it re-runs. [] = run once on mount. [x] = run when x changes. No array = run after every render. Never put a fetch directly in the component body.

Props

Arguments passed from parent to child. Read-only from the child's perspective. A child never modifies its own props. To have a child change parent data, the parent passes a callback function as a prop.

State Lifting

When two sibling components need the same data, move it to their common ancestor. The ancestor owns the state. Siblings receive it via props and update it via callbacks. "Lift state up" until the data lives in the right place.

Controlled Inputs

An input whose value is bound to React state and whose onChange updates that state. React owns the truth. The DOM is just a display. This enables validation, clearing, and syncing between inputs.

Immutability

React's re-render trigger is a new object reference. Mutating in place (.push(), direct assignment) doesn't change the reference, so React doesn't re-render. Always return new arrays and objects: spread [...arr], {...obj}.

Rendering Lists

.map() transforms an array into an array of JSX elements. Each element needs a unique, stable key prop on its outermost element. This key must come from your data, not from the array index.

Conditional Rendering

Three tools: early return for mutually exclusive states, the && operator for "render if true," and a ternary for "render A or B." Use early returns for loading/error/data — they read like plain English.

Derived State

If a value can be computed from existing state, compute it — don't store it in a separate useState. The running total is always expenses.reduce(...). Two sources of truth for the same fact always desync. Derive; don't duplicate.
